How Azure Application Groups
Organise and Govern Access

Application Groups are the core access-delegation primitive in Azure Virtual Desktop — and a useful pattern for broader workload governance. Here's how they work, why they matter, and how to deploy them.

Assigned Principals

As Azure environments grow in complexity, the question of who can access what becomes harder to answer cleanly. Azure Application Groups address this directly — they give you a named, auditable unit that bundles published resources and maps them to users, sitting neatly between your compute infrastructure and your end users.

Definition: An Application Group is an Azure resource that publishes either a full Remote Desktop or a set of individual RemoteApp applications to assigned users — all served from a Host Pool of session hosts.

What Problem Do They Solve?

Without Application Groups, access control in a virtual desktop environment degrades quickly. You end up either giving everyone a full desktop (overly broad), or managing per-user session host permissions manually (unmanageable at scale). Application Groups insert a clean abstraction layer: the resource definition lives in the group, and you assign Azure AD users or groups to it — nothing more required.

This maps naturally onto team structures. The Finance team gets the Finance Application Group, which publishes their ERP client and Power BI gateway. The Engineering team gets a separate group with their own toolset. Users see only what they're assigned.

Types of Application Groups

There are two types, and you can have multiple groups per Host Pool:

  • Desktop Application Group — publishes a full Windows desktop session. One per Host Pool. Best for users who need a general-purpose virtual workstation.
  • RemoteApp Application Group — publishes individual applications. Users see those apps in their Start menu or browser. Multiple RemoteApp groups can share a single Host Pool, which is the cost-efficient choice for specialised workloads.

Deploying with Azure CLI

Creating an Application Group is a three-step process: create the Host Pool, create the Application Group linked to it, then assign users.

# Step 1 — Create the Host Pool
az desktopvirtualization hostpool create \
  --resource-group myRG \
  --name myHostPool \
  --host-pool-type Pooled \
  --load-balancer-type BreadthFirst \
  --location eastus

# Step 2 — Create a RemoteApp Application Group
az desktopvirtualization applicationgroup create \
  --resource-group myRG \
  --name FinanceApps \
  --host-pool-arm-path /subscriptions/.../hostpools/myHostPool \
  --application-group-type RemoteApp \
  --location eastus

# Step 3 — Assign a user group
az role assignment create \
  --role "Desktop Virtualization User" \
  --assignee finance-team@contoso.com \
  --scope /subscriptions/.../applicationGroups/FinanceApps

Governance & RBAC Integration

Application Groups integrate directly with Azure RBAC. The Desktop Virtualization User built-in role is the correct assignment — it grants users permission to connect without any ability to modify infrastructure. Platform administrators keep the Desktop Virtualization Contributor role at the resource group level.

Assigning roles at the Application Group scope rather than the subscription means user access is narrowly scoped by design — satisfying least-privilege requirements without additional configuration.

Tagging for Cost Attribution

For organisations running multiple Application Groups, tagging is essential for accurate cost reporting. Apply a consistent schema at creation time:

{
  "ApplicationGroup": "FinanceApps",
  "Team":             "Finance",
  "Environment":     "Production",
  "CostCentre":      "CC-20041"
}

In Azure Cost Management, filter by the ApplicationGroup tag to produce per-team cost breakdowns. This is particularly useful in demo environments — stakeholders can see exactly what each Application Group costs, isolated from other resources.
